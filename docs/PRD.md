# 부서 피드백 관리 시스템 MVP PRD

## 핵심 정보

**목적**: 부서원들이 LLM/ERP 서비스에 대한 피드백을 익명으로 제출하고, 관리자가 이를 취합·관리할 수 있는 내부 웹앱 제공
**사용자**: 23명 규모 부서의 전 직원 (일반 사용자) + 관리자 1명 (최정인)

## 사용자 여정

```
1. 로그인 페이지 (로그인/자동가입 통합)
   ├── [기존 사용자] 이름/비밀번호 입력 → 로그인
   └── [신규 사용자] 이름/비밀번호 입력 → 비밀번호 확인 → 자동 가입 + 로그인

2. 피드백 목록 페이지 (기본: LLM 피드백 탭)
   ├── LLM 피드백 탭 ← 기본 선택
   │   ├── 피드백 안내 설명 확인
   │   ├── 피드백 목록 조회 (작성자 익명)
   │   ├── 피드백 검색
   │   ├── 엑셀 다운로드 (작성자 가림 / 관리자는 포함)
   │   ├── [작성] 버튼 → 피드백 작성 페이지
   │   └── 본인 작성건 [수정] → 피드백 수정 페이지
   │
   └── ERP 피드백 탭
       └── (LLM 탭과 동일 구조)

   [관리자 전용]
   └── 작성자 이름 확인 가능

3. 피드백 작성 페이지
   ↓ 피드백 내용 입력 후 저장
   → 피드백 목록 페이지로 이동

4. 피드백 수정 페이지 (본인 작성건만)
   ├── 수정 → 저장 → 피드백 목록 페이지
   └── 삭제 → 확인 → 피드백 목록 페이지
   
```

## 인증 설계

### 인증 방식

이름 + 비밀번호 기반 로그인/자동가입 통합 방식. 사용자에게 이메일은 노출되지 않으며, Supabase Auth 내부 구현 상세로만 사용된다.

| 항목 | 내용 |
|------|------|
| **사용자 입력** | 이름 + 비밀번호 |
| **인증 방식** | 로그인/자동가입 통합 (이름 기반 자동 판별) |
| **내부 구현** | Server Action 내부에서 Supabase Auth API 호출 (사용자에게 비노출) |

### 로그인/자동가입 통합 흐름

```
1. 사용자 입력: 이름, 비밀번호
2. 서버에서 이름 존재 여부 확인 (checkUserExists)
   - [기존 사용자] 비밀번호 검증 → 로그인
   - [신규 사용자] 비밀번호 확인 필드 표시 → 자동 가입 + 로그인
3. Auth 성공 시 → DB 트리거로 Users 테이블에 프로필 레코드 자동 생성
   - auth_id: auth.users.id
   - name: 입력한 이름
   - is_admin: 이름이 '최정인'이면 true, 아니면 false
4. 세션 토큰 발급 → 피드백 목록 페이지로 이동
```

### 관리자 설정

- 이름 `최정인`으로 가입 시 `is_admin = true` 자동 설정
- DB 트리거에서 메타데이터의 name을 추출하여 처리

## 익명성 보장 전략

### API 응답 설계

일반 사용자에게는 작성자 정보를 노출하지 않아 브라우저 개발자 도구로도 확인 불가능하게 한다.

| 사용자 유형 | 피드백 응답 필드 |
|------------|-----------------|
| **일반 사용자** | `id`, `category`, `content`, `created_at`, `updated_at`, `is_mine` (boolean) |
| **관리자** | 위 필드 + `author_name` (작성자 이름) |

- `is_mine`: 현재 로그인한 사용자의 작성건 여부 (본인 수정 버튼 표시용)
- `author_id`는 클라이언트에 **반환하지 않음** (서버에서 `is_mine` 플래그로 대체)

### Supabase RLS 정책 (참고)

```sql
-- 피드백 조회: 모든 로그인 사용자 가능 (작성자 정보는 서버에서 필터링)
-- 피드백 작성: 로그인 사용자만 가능
-- 피드백 수정/삭제: 본인 작성건만 가능
```
